# DashGen v0.2 — Plan

**Status**: draft, authored at the v0.1 launch gate (deterministic core proven
end-to-end against live Prometheus + Grafana). Supersedes nothing; complements
`SPECS.md` (v0.1 execution spec) and `ROADMAP.md` (v0.2 stage definition).

Read alongside `RECIPES.md`, which enumerates the recipe catalog this plan
depends on.

---

## 1. Product delta from v0.1

v0.1 proved that DashGen can generate deterministic, reviewable dashboards
from a single Prometheus endpoint. v0.2 keeps that core *exactly* as-is and
adds three capabilities:

1. **Optional AI enrichment** — explicitly opt-in, feature-flagged, cached
   by inventory hash, and structurally unable to bypass the deterministic
   validation pipeline.
2. **Recipe coverage expansion** — from the current 12 recipes (across
   service/infra/k8s) to ~35 (see `RECIPES.md`), driven by real-world
   metric shapes observed in broad Prometheus deployments.
3. **Quality tooling** — `lint`, `coverage`, and partial-regeneration so
   operators can run DashGen against existing dashboards and understand
   what's drifted or missing.

The v0.1 cross-stage rules from `ROADMAP.md` still govern everything:

> Deterministic first, AI second. Safe by default. Better to omit than
> invent. Reviewability is mandatory. Scope stays narrow.

If any v0.2 feature *starts* to contradict one of these, that feature
yields — not the rule.

## 2. AI scope

### 2.1 What AI may touch

| Layer | AI role | Deterministic path |
|-------|---------|--------------------|
| Classification of **unknown** metrics | Propose candidate traits (e.g., "looks like RPC latency"); always advisory | If AI unavailable or disabled, metric stays `TraitsUnknown` — no panel emitted. Omission beats invention. |
| Panel **titles** | Rewrite mechanical titles (`Request rate: api_http_requests_total`) into human-scannable ones (`API request rate per handler`) | Mechanical title is always preserved in the rationale as the canonical identifier. |
| **Rationale** prose | Enrich the per-panel "why" with domain context drawn from metric help text + labels + fleet shape | Mechanical sentence from the recipe is always emitted too, as a fallback and audit trail. |
| **Recipe drafting** (offline authoring tool) | Given a corpus of real metrics, propose new recipe Go code for a human to review and commit | The committed code is deterministic; AI output is a scaffolding tool that never runs in the dashboard generation path. |
| **Unknown-family grouping** | When the classifier cannot match a recipe, AI may cluster unknown metrics into a "custom" section with heuristic titling | Only fires for metrics that would otherwise be dropped by v0.1. Never overrides a deterministic match. |

### 2.2 What AI may NOT touch

- **PromQL generation.** All queries are synthesized by deterministic code
  in `internal/recipes/*` and then validated by `internal/validate/*`.
  The AI never produces a query string.
- **Verdicts.** `accept` / `accept_with_warning` / `refuse` are strictly
  the output of the validate pipeline. AI cannot upgrade a refused query
  or downgrade an accepted one.
- **Safety decisions.** Banned-label enforcement, cardinality checks, and
  strict-mode gating are deterministic. AI has no say.
- **Panel UIDs and dashboard UIDs.** Stable IDs come from
  `internal/ids` and hash deterministic inputs. AI output never enters
  the ID material.
- **Section ordering.** Profiles own this.
- **Runtime behavior during replay** — every enrichment call goes through
  the cache; a cache hit means zero network calls. Rerunning against the
  same inventory must produce byte-identical output.

### 2.3 Provider interface

```go
package enrich

type Enricher interface {
    // Describe the provider (name, model, offline/online) for rationale.
    Describe() Description

    // ClassifyUnknown proposes trait candidates for metrics the
    // deterministic classifier could not place. Returning an empty map
    // is always valid. No error returned unless the provider itself
    // failed — unknown metrics are fine.
    ClassifyUnknown(ctx context.Context, in ClassifyInput) (ClassifyOutput, error)

    // EnrichTitles proposes human-scannable titles. Must return the same
    // number of titles as were requested, in the same order. If it
    // cannot, it returns an error and the caller falls back.
    EnrichTitles(ctx context.Context, in TitleInput) (TitleOutput, error)

    // EnrichRationale proposes supplementary rationale paragraphs per
    // panel. Same ordering contract as EnrichTitles.
    EnrichRationale(ctx context.Context, in RationaleInput) (RationaleOutput, error)
}
```

**Initial providers** (per ROADMAP, scope adjusted in 2026-04-25):

- **Hosted (primary)**: `anthropic` backed by `claude-opus-4-7`. Explicit
  opt-in via `--provider anthropic` plus `ANTHROPIC_API_KEY`. Never the
  default.
- **Hosted (alternate)**: `openai` backed by `gpt-5` or equivalent.
  Same opt-in shape, requires `OPENAI_API_KEY`.
- **Local provider**: deferred to v0.3 backlog. The previous plan named
  ollama as the local default; that work has been removed from v0.2 scope
  to keep AI surface area small while two hosted providers are
  hardened. The `Enricher` interface and the `enrich.New(Spec)` provider
  registry remain shaped for a local provider drop-in (see §2.7) so the
  v0.3 work is a one-file change.

### 2.7 Provider extension contract

`internal/enrich/factory.go` is the single extension point for new
providers. To add one — local or hosted, today or in the future — a
contributor:

1. Creates `internal/enrich/<name>.go` with a Constructor of type
   `func(Spec) (Enricher, error)` that builds the concrete enricher.
2. Calls `enrich.Register("<name>", ctor)` from that file's `init()`.

Nothing else changes. `internal/app/generate.applyEnrichment` already
calls `enrich.New(Spec{Provider: cfg.Provider, ...})` and dispatches by
the returned enricher's `Describe()`. CLI surface accepts any string
that resolves through the registry.

`Spec` carries the cross-provider knobs (`Provider`, `Model`, `CacheDir`,
`NoCache`). Provider-specific configuration (API keys, retry tuning,
local socket paths) belongs inside the Constructor — typically read from
environment variables — so adding a provider does not balloon `Spec` or
the public CLI surface.

Two error sentinels distinguish failure modes for callers:

| Sentinel | Meaning |
|----------|---------|
| `ErrUnknownProvider` | The provider name has no registration. CLI / config errors map here. |
| `ErrNotImplementedYet` | The provider name is registered as a placeholder (anthropic, openai, ollama, …) but has no working implementation in this build. Distinct from "unknown" so a misconfigured environment is debuggable from the error chain. |

#### Soft-fail policy

`applyEnrichment` calls each requested dispatch mode (`EnrichTitles`,
`EnrichRationale`) independently. On enricher error, `applyEnrichment` logs
to stderr (`enrich titles: <err>` / `enrich rationale: <err>`) and skips that
mutation; subsequent modes are still attempted. The dashboard returned is the
deterministic mechanical output — unchanged. AI failures never crash a run or
produce partial output.

### 2.4 Caching

All enrichment results are cached under `~/.cache/dashgen/enrich/<inventory-hash>/<function>.json`.

- Inventory-hash is computed the same way as the dashboard UID (see
  `internal/inventory/InventoryHash`).
- A cache hit bypasses the network entirely. Two runs over the same
  inventory produce byte-identical output.
- `--no-enrich-cache` forces re-fetch, intended only for authoring /
  debugging.
- Cache entries include `provider_id`, `model_id`, `dashgen_version`, and
  `prompt_hash` so evolving prompts don't silently reuse stale answers.

### 2.5 Failure modes and defaults

| Condition | Behavior |
|-----------|----------|
| No `--provider` flag | Deterministic path only; no enrichment attempted. |
| Provider configured but unreachable | Log at `warn`, continue deterministic-only. Non-fatal. |
| Provider returns malformed output | Log, discard the batch, continue deterministic. |
| Provider returns output that fails validation | Validation pipeline refuses as usual. |
| `--strict` + enrichment output has warnings | Strict semantics apply uniformly — same as deterministic warnings. |

### 2.6 CLI surface additions

| Flag | Description |
|------|-------------|
| `--provider {off,anthropic,openai}` | Pick enrichment provider. Default `off`. Any name registered in `internal/enrich/factory.go` is accepted; unknown names fail fast with `ErrUnknownProvider`. |
| `--provider-model <id>` | Override the default model for the provider. |
| `--enrich {titles,rationale,classify,unknown-grouping,all,none}` | Comma-separated subset of enrichment operations to run. Default `none` when provider is `off`, `titles,rationale` when provider is set. |
| `--no-enrich-cache` | Force re-fetch. |
| `--cache-dir <path>` | Override cache location. |

```
dashgen generate \
  --prom-url http://prometheus:9090 \
  --profile service \
  --provider anthropic \
  --enrich titles,rationale \
  --out ./dashboards
```

`lint` and `coverage` are new top-level commands; their design lives in
`ROADMAP.md` §Stage 3, not here.

| Profile | v0.1 recipes | v0.2 Tier-1 additions | v0.2 Tier-2 additions | Total shipped |

Kafka, RabbitMQ, SMART disk, hardware telemetry, CNI) are deliberately
deferred.

## 4. Phased delivery

### Phase 1 — Foundation (no user-visible AI)

Work that must land first so later phases can assume these exist:

1. `internal/enrich` package with the `Enricher` interface and a
   `NoopEnricher` that always returns empty output. `generate.Run`
   grows an optional enricher parameter; default remains nil → no
   behavior change.
2. Cache layer with inventory-hash keying. Tests for hit/miss, TTL,
   corruption recovery.
3. Help-text → trait-hint extraction in `internal/classify` (no AI yet;
   deterministic substring + regex). This is the groundwork the AI
   phase then *extends* rather than *replaces*.
4. Recipe authoring contract tightening: every recipe gains a
   `MatchFunc`-style test, a BuildPanels test, a fixture that exercises
   it, and a discrimination test ensuring it doesn't match off-shape
   metrics. Backfill this onto existing v0.1 recipes first
   (service_http_rate still has no dedicated discrimination test; the
   v0.1 `service-realistic` fixture now covers errors and latency but
   not the rate recipe's false-positive surface).

**Phase-1 exit**: `make test` still fully deterministic; recipe catalog
covers Tier-1 for at least one profile end-to-end; no AI code paths yet.

### Phase 2 — Recipe catalog expansion

Execute the recipe catalog per `RECIPES.md`, one profile at a time:

1. **service**: gRPC family, Go runtime family, client_http, db_pool,
   tls_expiry.
2. **infra**: load, filesystem_usage, fd_usage, nic_errors.
3. **k8s**: deployment_availability, node_conditions, pvc_usage,
   oom_kills.

Each recipe ships with: Match test, BuildPanels test, fixture entries,
regression golden. Acceptance for the phase is the Tier-1 list in
§3 having 100% coverage.

### Phase 3 — Hosted AI (anthropic, primary)

**Dispatch integration: ✅ DONE (commit 1c2f298)** — `EnrichTitles` and
`EnrichRationale` are wired in `applyEnrichment`; `ClassifyUnknown` gating
below remains the Phase 3 boundary.

1. Implement `AnthropicEnricher` behind the `Enricher` interface,
   registered via `enrich.Register("anthropic", ...)` in
   `internal/enrich/anthropic.go`.
2. Wire `EnrichTitles` first. Lowest risk — even if the model
   hallucinates, the deterministic title is preserved in the rationale
   and the rendered dashboard still works.
3. Wire `EnrichRationale` second. Same determinism guarantees.
4. Wire `ClassifyUnknown` third. This is the riskiest of the three; it
   affects whether a recipe fires, so the surface is carefully gated: the
   AI's trait assignment is only consulted for metrics the deterministic
   classifier left as `MetricTypeUnknown` *and* `traits=[]`.
5. `--provider anthropic` becomes a documented opt-in flag; cache
   behavior validated (second run with cache, byte-identical to first).
6. Redaction: no label *values* ever sent, only label *names* + metric
   names + metric help text. Values can contain PII or internal IDs.
   Verified by a proxy-capture test (per §6 acceptance).
7. Clear logging of what gets sent to the provider (pre-send summary,
   byte count, approximate token budget; governed by a
   `--log-enrichment-payloads` flag for debugging).
8. Docs call out the network boundary explicitly.

### Phase 4 — Hosted AI (openai, alternate)

**Dispatch integration: ✅ DONE (commit 1c2f298)** — `OpenAIEnricher` registered;
`EnrichTitles` and `EnrichRationale` dispatch confirmed. One-file addition
(`internal/enrich/openai.go`) validates the registry contract.

1. `OpenAIEnricher` implementation following the exact same shape as the
   anthropic enricher; registered via `enrich.Register("openai", ...)`
   in `internal/enrich/openai.go`. Re-uses the redaction helper, prompt
   templates, and cache-key derivation from Phase 3 — Phase 3 is the
   hardening pass, Phase 4 is the second-provider mirror.
2. Same redaction guarantees, same `--log-enrichment-payloads` debug
   path, same opt-in shape (`OPENAI_API_KEY`).
3. Reaffirms the contract: the provider registry pattern means a
   second hosted provider is a one-file change. If Phase 4 needs more,
   that signals the contract is wrong and the registry must be revised
   before more providers ship.

### Phase 4.5 — Local provider (deferred to v0.3 backlog)

The earlier draft of this plan made `ollama` the Phase 3 entry. That
work is now deferred:

- The `Enricher` interface and the `enrich.New(Spec)` registry stay
  shaped for a local provider — adding one is a single file in
  `internal/enrich/` plus a one-line `Register` call.
- A registered placeholder (`ollama`) returns `ErrNotImplementedYet`
  today, signalling to operators that the boundary is shaped but the
  body is missing. Operators who need offline AI can implement against
  the same interface.
- v0.3 picks up the local provider work alongside `/metrics` input,
  cross-section partial regeneration, and Tier-3 recipes.

### Phase 5 — Unknown-family grouping

The most ambitious AI feature: cluster unmapped metrics into a
"custom" section.

1. Input: metrics with no deterministic trait match.
2. AI clusters by semantic similarity, proposes section names.
3. Each proposed panel is still a deterministic query (counter → rate,
   gauge → raw, histogram → quantile). Only the *grouping* is AI-driven.
4. `--enrich unknown-grouping` must be opt-in separately from
   titles/rationale because it *does* create new panels; if the AI is
   wrong about the grouping, the user sees unexpected panels.
5. Strict-mode behavior: `--strict` refuses the run if any AI-proposed
   grouping produces a warning.

### Phase 6 — `lint` + `coverage`

Separate tracks; can parallelize with Phase 5. Design lives in
`ROADMAP.md`.

## 5. Dependencies on v0.1

The following v0.1 behaviors are *load-bearing* for v0.2 and must not
regress:

- Inventory hash is stable and a pure function of the inventory.
  (Enrichment caching keys off this.)
- Dashboard / panel UIDs are deterministic. (AI-enriched output still
  uses these as canonical identity so re-enrichment lines up.)
- Validation pipeline stages 1-5 run on every query regardless of
  origin. (AI never bypasses.)
- Strict mode semantics are clean: any warning ⇒ refuse.
- Renderer is fully separated from synthesis. (AI output feeds the IR
  exactly like a recipe would; the renderer does not branch on
  "enriched or not".)

A v0.2 change that would require violating any of these must be
escalated: update `SPECS.md` and `ROADMAP.md` first.

## 6. Acceptance criteria for v0.2

| Area | Must pass |
|------|-----------|
| Determinism | Two generate runs over the same inventory with the same provider + cache produce byte-identical `dashboard.json`, `rationale.md`, `warnings.json`. |
| AI-off parity | `--provider off` over the v0.1 golden fixtures produces exactly the v0.1 goldens (regenerated once for any *deterministic* change, not for AI). |
| AI-on repeatability | With a populated cache, `--provider anthropic` (or any hosted provider) produces byte-identical output to the previous run, no network calls fired (verified by mock proxy). |
| Recipe coverage | Tier-1 recipes from `RECIPES.md` all have code, tests, fixtures, and at least one non-regressing golden. |
| Discrimination | Each recipe ships with at least one negative-case fixture metric that would naively trip one signal. Test must fail if the recipe matches it. |
| Safety | No AI path emits a query. No AI path can upgrade a refused verdict. |
| Redaction | Hosted provider requests contain zero label values and zero instance endpoints. Verified by a proxy-capture test. |
| Strict mode | `--strict --provider <x>` refuses on any AI-introduced warning. |

---

## 7. Risks

| Risk | Mitigation |
|------|------------|
| AI mode silently becomes the default and trust erodes | `--provider off` is hard-coded default; CI runs with `off`; docs lead with deterministic examples. |
| Cache drift across dashgen versions | Cache keys include `dashgen_version` and `prompt_hash`. |
| Unknown-family grouping hallucinates a panel that's trivially wrong | Strict mode refuses by default; warnings surface AI-origin panels separately. |
| Recipe catalog expansion accidentally touches the v0.1 discrimination surface | Every recipe PR must include the discrimination-test pattern from `TestDiscrimination_ServiceRealistic`. |
| Help text is unreliable across exporters | Help-text signals are weighted last in the trait decision; labels dominate. |
| Live-backend fixture rot (demo.prometheus.io metrics change shape) | Committed fixtures are the authoritative regression set; live runs are ad-hoc validation only. |

## 8. Out of scope for v0.2 (reaffirmed)

- Always-on daemon
- Kubernetes operator
- Automatic `grafana apply`
- Alert generation
- SLO generation
- Incident diagnosis
- Multi-tenant control plane
- Training fine-tuned models on user data

These land in v0.3+ or never. Anything on this list that "leaks into"
v0.2 work triggers scope review.
