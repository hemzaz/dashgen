## 0. RALPLAN-DR Summary

### 0.1 Principles (3-5)

1. **Deterministic first, AI second** (SPECS §2.1; ROADMAP cross-stage rule 1). AI never generates PromQL, never flips a verdict, never produces cache-key material, never writes through directly. If an AI path *can* theoretically regress the v0.1 core, the plan step that introduces it also names the regression test that proves it didn't.
2. **Better to omit than invent** (SPECS §2.3; V0.2-PLAN §2.1 unknown-metric row). A failed provider call, a malformed response, or a refused query yields silence or deterministic fallback — never a plausible-looking filler.
3. **Refusals beat warnings when the panel shouldn't exist** (ADVERSARY §6 drift pattern "warnings instead of refusals"). Lint, unknown-family grouping, and strict mode refuse rather than warn when intent is "this dashboard element should not ship."
4. **Reviewability is non-negotiable** (SPECS §2.4). Every AI-origin title/rationale/grouping ships alongside the mechanical version in the rationale as an audit trail; every refusal has a specific, operator-readable reason; diffs stay minimal.
5. **Scope stays narrow** (ROADMAP cross-stage rule 5; ADVERSARY §11 "we need this abstraction for future providers"). Two providers, not a plugin system. No alerting, no SLO, no daemon, no `/metrics` mode leak, no multi-backend.

### 0.2 Decision Drivers (top 3)

1. **v0.1 determinism is load-bearing** (V0.2-PLAN §5). Every unlock we build is graded against "two runs over the same inventory produce byte-identical output." If an AI phase threatens this, it yields.
2. **Operator trust is finite and pre-consumed** (ADVERSARY §3 "safe by default"). The order we ship features in decides whether users meet DashGen's v0.2 flavor first as "it's still boring and safe" or as "it's doing AI stuff now." The former preserves trust; the latter risks burning it before lint/coverage even land.
3. **Execution bandwidth is constrained** (roughly 1 engineer + agentic assist, per recent cadence). Sequencing must front-load low-risk, high-leverage work so review fatigue doesn't accumulate on the AI-wiring PRs, which are the ones that most need careful review.

### 0.3 Viable Options

**Option A — "Harden before enrich"**.
Ship order: Phase 2 closure (5 stragglers + Tier-2 fixture extensions) → Phase 1 wiring closure (help-text hints, discrimination backfill) → Phase 6 (`lint` + `coverage` + partial regen) → Phase 3 (provider registry + anthropic) → Phase 4 (openai) → Phase 5 (unknown-family grouping). Enrich seams stay wired to `NoopEnricher` through Phase 2 and Phase 6. *(Original R1 phrasing named ollama as the Phase-3 entry; R2 rerouted to anthropic + openai. The ship-order shape is unchanged.)*

- Pros:
  - v0.1 core surface stays under continuous regression coverage the whole time.
  - Lint/coverage are *deterministic-only* code, so they land without ever touching the AI blast radius.
  - When the first AI code lands (Phase 3 anthropic per R2), lint + coverage + partial regen are already there to inspect enriched dashboards, and the Tier-2 fixture surface is there to prove enrichment didn't change deterministic output.
  - Matches ADVERSARY §6 drift pattern "broad scaffolding before real behavior" — we don't stand up an AI harness before we need it.
- Cons:
  - Users curious about AI wait longer. (Low cost — zero users today; OSS hasn't shipped v0.2.)
  - The first hosted enricher (anthropic per R2; ollama in the original draft) is behind 3 phases of non-AI work; a protocol change in the `Enricher` interface discovered late forces a larger blast-radius refactor.

**Option B — "Enrich first, tools later"**.
Ship order: Phase 3 (first hosted enricher) + wire into `generate.Run` with `--provider <first>` and `--enrich titles,rationale` → Phase 4 (second hosted enricher) → Phase 2 closure → Phase 1 wiring closure → Phase 5 → Phase 6. *(R2: first hosted = anthropic; second = openai. R1 draft used ollama as the first.)*

- Pros:
  - Dogfoods the AI seam early; catches `Enricher` interface mistakes before they calcify.
  - Phase 3/4 have the most engineering novelty, so front-loading burns risk early.
- Cons:
  - Recipes `service_request_size`, `service_response_size`, `infra_interrupts`, `k8s_scheduler_latency`, `k8s_coredns` have been "best-effort" for two batches; deferring them risks them never landing in v0.2.
  - AI PRs land against a Tier-2 catalog with known fixture gaps, so any "why did that title change?" question has the confounder "maybe the fixture is the wrong shape."
  - Higher reviewer load: reviewer has to hold "AI correctness" and "recipe correctness" simultaneously.

**Option D — "Noop-wire-first, harden-second, AI-third, tools-last"** (recommended).
Ship order: Step 0.5 (new; IR + config + `applyEnrichment` glue wired to `NoopEnricher`) → Phase 2 closure → Phase 1 wiring closure → Phase 6 (`lint` + `coverage` + partial regen) → Phase 3 (provider registry + first hosted enricher, including Step 4.0b regression suite against the already-wired Noop path) → Phase 4 (second hosted enricher) → Phase 5 (unknown-family grouping). *(R2: first = anthropic, second = openai; R1 draft = ollama then anthropic.)*

- Pros:
  - **Every subsequent deterministic step becomes free regression coverage for the `NoopEnricher` seam.** Steps 1.0-1.4, step 1.5 (fixtures), step 2.0 (help-text hints), step 2.1 (discrimination backfill), step 3.0 (lint), step 3.1 (coverage), step 3.2 (idempotent regeneration) all regenerate goldens through a pipeline that already contains `applyEnrichment`, so any mistake in the glue surfaces on the very next step — not on commit 12 when Phase 3 finally lands.
  - **`Enricher` interface contract mistakes surface on commit 2 instead of commit 12.** If `Panel.MechanicalTitle` / `Panel.RationaleExtra` have the wrong shape, or `applyEnrichment` is sequenced wrong relative to `validateAndFinalize`, or `NoEnrichCache` / `CacheDir` semantics are wrong, the Tier-2 recipe PRs catch it immediately because their goldens re-run through that path.
- Cons:
  - Forces IR contract change (`Panel.MechanicalTitle`, `Panel.RationaleExtra`) before any recipe work begins.
  - Raises the floor on "what must exist" for Phase 2 work: the first Tier-2 recipe PR can no longer land against a v0.1-shaped IR.

**Option C — "Lint/coverage alongside AI"** (invalidated).
Parallelize Phase 6 with Phase 3 so the AI-generated dashboards can be immediately linted.
*Why invalidated*: Phase 6 touches `internal/app/lint/`, `internal/app/coverage/`, and partial-regen which lands in `internal/synth/` + `internal/ids/` — those two changes share zero code with `internal/enrich/`. The only argument for parallelism is calendar time, and per Driver 3 we don't have the review bandwidth to run two semi-independent tracks simultaneously without the AI track consuming all the attention and lint/coverage landing under-reviewed. Options A and D both capture the same work without that risk.

**Decision**: **Option D**.

Rationale for choosing D over A: A's single material con — "The first hosted enricher is behind 3 phases of non-AI work; a protocol change in the `Enricher` interface discovered late forces a larger blast-radius refactor" — is exactly what Option D eliminates. D pays a small, one-commit tax (Step 0.5: two IR fields, four config fields, one glue function, a passthrough `NoopEnricher` call site) in exchange for turning every subsequent deterministic commit into regression evidence for the enrichment seam. The two cons of D (forced IR contract change up front; raised floor for Phase 2) are bounded and surface-level: the IR change is additive (zero-value defaults preserve byte-identical v0.1 output) and the "raised floor" is a single commit any downstream PR rebases on. Under Principle 1 ("if an AI path can theoretically regress the v0.1 core, the step that introduces it also names the regression test"), D lets us name that regression test at step 0.5 — *commit 2* — instead of step 4.0 — *commit 12* — which is strictly the stronger posture.

## 9. Ship-Order Recommendation

Ship in Option-D order: **noop-wire-first, harden-second, AI-third, tools-last**. Step 0.5 lands `NoopEnricher` as the pipeline default at commit 2, so every deterministic step that follows (Phase 2, Phase 1-closure, Phase 6) regenerates goldens through a pipeline that already contains `applyEnrichment` — making each of those steps incidental regression coverage for the enrichment seam. When the first real AI provider lands in Phase 3, lint + coverage + idempotent-regen already exist and the seam has already been exercised by 10+ commits' worth of determinism work.

### 9.1 ADR

- **Decision**: Sequence v0.2 remainder as **Option D**: Step 0.5 (`NoopEnricher` wired at commit 2) → Phase 2 closure → Phase 1 wiring closure → Phase 6 (lint/coverage/idempotent regen) → Phase 3 (provider registry + Anthropic, including 4.0b regression suite) → Phase 4 (OpenAI as second hosted provider) → Phase 5 (unknown-family grouping). Cut `/metrics` input from v0.2 scope; defer to v0.3. Defer cross-section partial regeneration to v0.3. Defer local AI provider to v0.3 (registry placeholder shipped so the drop-in is a one-file change). **Plan-revision 2026-04-25**: original sequencing made ollama the Phase 3 entry; commit `3591d41` removed local AI from v0.2 scope and Phases 3/4 swap accordingly.
- **Drivers** (top 3): (1) v0.1 determinism is load-bearing; (2) operator trust is finite and pre-consumed; (3) execution bandwidth requires low-risk, high-leverage work up front.
- **Alternatives considered**:
  - *Option A (harden before enrich — wire Enricher only in Phase 3)*. Rejected: concentrates all `Enricher` interface risk onto the single Phase 3 PR; a protocol-shape mistake discovered there forces blast-radius refactor across already-committed Tier-2 goldens. Option D captures Option A's safety posture and adds free regression coverage for the seam on commits 2-11.
  - *Option B (enrich first, tools later)*. Rejected: front-loads AI work against a Tier-2 catalog with fixture gaps, raising reviewer load during the highest-novelty PRs.
  - *Option C (lint/coverage alongside AI)*. Invalidated: Phase 6 and Phase 3 share no code; parallelism only buys calendar time, at the cost of attention-splitting on the AI track.
- **Why chosen**: Option D is the only option that keeps the v0.1 core under byte-identical regression coverage across *every* deterministic change AND turns those deterministic changes into incidental regression evidence for the enrichment seam, while still shipping the AI surface v0.2 promises. The single con — forced IR contract change up front — is bounded (two additive fields with zero-value defaults) and surfaces any contract mistake on commit 2 instead of commit 12.
- **Consequences**:
  - Users wait longer for AI features (zero cost; pre-release).
  - `Enricher` interface edits discovered in Phase 3 are trivially rebased because the wiring was already committed at Step 0.5 and every deterministic step since has been consuming it.
  - Phase 6 lint command will need one pass of follow-up when AI-enriched dashboards start existing (Phase 3), since lint may need to distinguish AI-origin from mechanical titles. This is acceptable scope.
  - Cross-section partial regeneration is explicitly out of scope for v0.2; operators using `--in-place` get idempotent-on-unchanged-inventory and full-rewrite-on-changed-inventory.
- **Follow-ups**:
  - v0.3 planning: pick up `/metrics` input, real partial regeneration (with `ids.DashboardUID` rework), Tier-3 recipes, CI-native workflow, and the Grafana-apply question. RECIPES.md §5 is the starting catalog.
  - After Phase 3 ships: audit whether `--enrich=titles` alone is already most of the operator value, which would make `rationale`/`classify` lower-priority in the v0.2.1 follow-up.
  - Whenever prompt templates in `internal/enrich/prompts.go` change, include an explicit cache-invalidation note in the commit message so operators running `dashgen` on cached environments know to clear cache.
