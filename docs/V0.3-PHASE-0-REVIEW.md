## 5. WARN findings (non-blocking)

**W1 (hardening) — `#MetricNameASCII` accepts non-printable ASCII.** `schema.cue:45` regex `^[\x00-\x7f]+$` allows BEL (0x07), NUL (0x00), DEL (0x7F), and other control bytes. Live probe with `name_equals: "foo<BEL>bar"` PASSES unification. The schema's stated T20 scope is Cyrillic-style homoglyphs and the byte-range is sufficient for that, but the Prometheus exposition format is `[a-zA-Z_:][a-zA-Z0-9_:]*`. **Recommendation:** Phase 1A T1A.1 loader adds a printable-ASCII check (or the stricter Prometheus name-shape) as defense-in-depth — schema's seven name-shape fields are partial-match (prefix/suffix/contains), so they cannot use the strict full-name regex; the loader is the right layer. Not filed as a fix task; flagged here for T1A.1 awareness.

**W2 (doc drift) — DSL §4.1 + §4.5 show un-nested identity fields.** The spec's CUE prose still places `name`/`section`/`profile`/`confidence`/`tier` at the top level of `#Recipe`, but the actual schema (commit f917c01) and the wire format (DSL §5.1) nest them under `metadata: #Metadata`. Schema is correct; §4 prose is stale. **Filed:** task #5 (`docs(v0.3): fix DSL §4 / §4.5 to use metadata-nested identity fields`). Blocks Phase 8 release-prep, not Phase 1A.
